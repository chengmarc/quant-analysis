# Cryptocurrency Quantitative Analysis

A collection of quantitative analysis tools for cryptocurrency markets, covering data extraction, market statistics, on-chain analysis, price simulation, and algorithmic trading.

---

## Subprojects

### 1. Market Overview (`overview/`)

A full data pipeline built around a historical snapshot of ~28,000 cryptocurrencies sourced from CoinGecko. The extraction stage uses a Selenium-based scraper to batch-download price series across all listed assets.

**Distribution Fitting**

For each asset, daily percentage returns are computed and a best-fit parametric distribution is selected by minimizing the Kolmogorov-Smirnov statistic over a candidate set:

$$D_n = \sup_{x} \left| F_n(x) - F(x) \right|$$

where $F_n$ is the empirical CDF of observed returns and $F$ is the CDF of a candidate distribution. The candidate set includes Normal, Cauchy, Student's $t$, F, Alpha, Beta, Gamma, Chi, and Chi-squared families.

**Cross-Sectional Market Statistics**

For each date $t$, let $\{x_i^{(t)}\}_{i=1}^{N}$ denote the log-scale market capitalizations (or trading volumes) of all active coins. The following statistics are tracked over time:

$$\mu_t = \frac{1}{N}\sum_{i=1}^N x_i^{(t)}, \qquad \sigma_t = \sqrt{\frac{1}{N}\sum_{i=1}^N \left(x_i^{(t)} - \mu_t\right)^2}$$

along with the 10th, 25th, 50th, 75th, and 90th percentiles of the cross-sectional distribution.

**Monte Carlo Price Simulation**

Fitted distributions are used to simulate one-year price paths. Each path is generated by iteratively sampling daily returns:

$$P_{t+1} = P_t \cdot \left(1 + \frac{r_t}{100}\right), \qquad r_t \sim \hat{F}$$

where $\hat{F}$ is the fitted distribution for the asset. Returns are rejection-sampled to $r_t \in (-30\%, +30\%)$ to suppress extreme tail artifacts. 100 independent paths are generated per asset.

### 2. Price Simulation (`random_walk.py`, `random_walk_interval.py`)

Two Monte Carlo simulation approaches for Bitcoin price forecasting.

**Stationary simulation** (`random_walk.py`) fits a single distribution to the full historical daily return series and samples forward unconditionally.

**Halving-cycle simulation** (`random_walk_interval.py`) partitions the historical return series into 30-day windows relative to each halving event. For interval position $k$, returns from the same relative window across **all historical cycles** are pooled together:

$$\mathcal{D}_k = \bigcup_{c} \{ r_t : t \in [\tau_c + (k-1)\Delta,\; \tau_c + k\Delta) \}$$

A single distribution $\hat{F}_k$ is then fitted to $\mathcal{D}_k$. This pooling assumption treats each halving cycle as a noisy draw from a shared underlying return process at each relative phase, increasing sample size at the cost of assuming stationarity across cycles. Forward simulation chains these interval-specific distributions sequentially:

$$r_t \sim \hat{F}_k, \qquad t \in \bigl[(k-1)\Delta,\; k\Delta\bigr) \text{ days after forecast start}$$

### 3. On-Chain Analysis (`someplots.py`, `UTXO.ipynb`)

Analysis of Bitcoin network fundamentals using two independent data sources.

`someplots.py` uses CoinMetrics community on-chain data merged with Yahoo Finance price data. `UTXO.ipynb` operates on the Google BigQuery public dataset (`bigquery-public-data.crypto_bitcoin`), querying raw UTXO inputs and outputs directly from the Bitcoin ledger.

Key metrics include:

**Annualized dilution rate**

$$\delta_t = 100 \cdot \frac{365 \cdot \overline{I}_{t,30}}{S_t}$$

where $\overline{I}_{t,30}$ is the 30-day rolling average of daily issuance and $S_t$ is total supply.

**Monetary velocity**

$$V_t = 100 \cdot \frac{\overline{\text{Vol}}_{t,90}}{\text{MarketCap}_t}$$

where $\overline{\text{Vol}}_{t,90}$ is the 90-day rolling average of total daily transaction volume.

**NVT ratio** (Network Value to Transactions)

$$\text{NVT}_t = \frac{\text{MarketCap}_t}{\text{TxVol}_t}$$

A high NVT indicates the network is being valued well above its current transactional utility — analogous to a P/E ratio for on-chain activity. In practice, the implementation uses the `NVTAdj90` field provided by CoinMetrics, which applies a 90-day smoothed adjustment to transaction volume.

**Metcalfe-based valuation**

Under Metcalfe's Law, network value scales with the square of active participants. A price-to-Metcalfe ratio $\text{PM}_t = P_t / N_t^2$ is computed, where $N_t$ is the daily active address count. Sustained deviation of $\text{PM}_t$ from its historical mean is interpreted as over- or under-valuation relative to network usage.

**Bitcoin P/E ratio**

$$\text{P/E}_t = \frac{\text{MarketCap}_t}{365 \cdot R_{t}}$$

where $R_{t}$ is the daily miner revenue (block rewards + transaction fees), annualized by multiplying by 365.

### 4. Binance Trading Framework (`binance/`)

A modular live trading system targeting the Binance REST API.

**Authentication**

All signed endpoints require an HMAC-SHA256 signature over the query string:

$$\text{signature} = \text{HMAC-SHA256}(K_{\text{secret}},\; \text{queryString} \,\|\, \text{requestBody})$$

Authentication is delegated to the `binance-connector` SDK; the `Connector` module wraps SDK calls to provide a consistent interface for market data retrieval.

**Trend Pattern Probability Matrix**

The `Algorithm` module encodes recent price trends as binary sequences $p = (d_1, d_2, \ldots, d_n) \in \{+1, -1\}^n$ and computes empirical transition probabilities from historical data:

$$\hat{\Pr}(\text{up} \mid p) = \frac{\bigl|\{t : \text{trend}_{t-n:t} = p,\; d_{t+1} = +1\}\bigr|}{\bigl|\{t : \text{trend}_{t-n:t} = p\}\bigr|}$$

Patterns are bitwise-encoded as integers for compact representation. The full probability matrix is computed for all patterns up to a configurable length, and is used by the `Strategy` module to generate trading signals.

### 5. Exponential DCA (`exponential-DCA/`, `exponential_DCA_stock.py`)

A signal-conditioned dollar-cost averaging strategy driven by a volatility-scaled log-residual signal.

**Log-trend decomposition**

The log price series is first decomposed into a logarithmic trend and a residual:

$$\log P_t = \underbrace{a \cdot \log(t + c) + b}_{\text{trend}} + \epsilon_t$$

where $(a, b, c)$ are estimated via non-linear least squares. The residual is then normalized by a linearly fitted volatility trend $\hat{\sigma}_t$:

$$z_t = \frac{\epsilon_t}{\hat{\sigma}_t} = \frac{\log P_t - \widehat{\log P}_t}{\hat{\sigma}_t}$$

**Signal rules**

Trading signals are generated from the scaled residual $z_t$:

$$\text{Signal}_t = \begin{cases} \text{Buy} & \text{if } z_t < -5 \\ \text{Sell} & \text{if } z_t > 10 \\ \text{Empty} & \text{otherwise} \end{cases}$$

The asymmetric thresholds reflect the empirical distribution of Bitcoin residuals: downside deviations are sharper and shorter-lived, while upside deviations are more gradual and extended.

**Allocation rule**

On each Buy signal, a fraction $\alpha$ of accumulated cash is deployed; on each Sell signal, a fraction $\alpha$ of current position is liquidated:

$$\Delta \text{position}_t = \begin{cases} +\,\alpha \cdot C_t \,/\, P_t & \text{if Signal}_t = \text{Buy} \\ -\,\alpha \cdot H_t & \text{if Signal}_t = \text{Sell} \end{cases}$$

where $C_t$ is cash balance and $H_t$ is current holdings. Both $\alpha$ and the start date are optimized via grid search. A full writeup is available in `exponential-DCA/Exponential DCA.pdf`.
